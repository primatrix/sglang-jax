## 剩余问题: 切桶 first-seen 详解

### 现象
PHASE4.5b refresh 数据:
```
[PHASE4.5b refresh] draft_fwd=0 gather=0 topk=0 slice=3   ← 3 个 slice trace
[PHASE4.5b refresh] draft_fwd=0 gather=0 topk=0 slice=1
```

closest-key 报告(bsz sweep 后):
```
forward_batch[12][3], now i32[7] and before i32[8]
forward_batch[12][3], now i32[3] and before i32[4]
forward_batch[12][3], now i32[15] and before i32[16]
```

### 根因
`capture_for_decode` 末尾 (eagle_draft_worker.py:378-380) 和 `draft_extend_for_decode` 内 (line 339-345) 用:
```python
topk_p = jax.lax.dynamic_slice_in_dim(topk_p, 0, real_bs, axis=0)
topk_index = jax.lax.dynamic_slice_in_dim(topk_index, 0, real_bs, axis=0)
hidden_states = jax.lax.dynamic_slice_in_dim(hidden_states, 0, real_bs, axis=0)
```

**`size=real_bs` 在 `dynamic_slice_in_dim` 是 Python int / static argument**, 每个 unique real_bs 触发一次 trace。同桶内 real_bs ∈ {5,6,7,8} 各 trace 1 次 × 3 个 slice (topk_p/topk_index/hidden_states) = 12 次 first-seen per bucket。

slice 必须保留是 **Phase 3.6 教训** (代码已注释): capture 输入保持 padded (命中 precompile bucket) → 输出 slice 到 real_bs (scheduler concat 不被 padding 行 inflate)。

### 为什么 Phase 4.9 尝试失败
试过给 `EagleDraftInput.allocate_lens` host-pad + restore。但**`allocate_lens` 不在 `EagleDraftInput.tree_flatten` children 里** (它是 dataclass 字段但不是 pytree leaf),不进 jit cache key,改它无效。已 revert。

## Option B 实施方案 (新线程接手)

### 目标
让 `capture_for_decode` 出口 **保留 padded shape**,scheduler 端识别 padding 行不要 inflate。

### 修改清单

#### 1. `eagle_draft_worker.py:capture_for_decode` (line 364-388)
去掉末尾的 `dynamic_slice_in_dim`,让输出保持 padded:
```python
def capture_for_decode(self, logits_output, draft_input, real_bs=None):
    topk_p, topk_index = topk_probs_from_logits(logits_output.next_token_logits, self.topk)
    hidden_states = replicate_to_mesh(self.mesh, logits_output.hidden_states)
    # Phase 4.10: drop the real_bs slice; output stays at padded_bs so the
    # next tick's draft/refresh JIT sees a stable input shape. Scheduler
    # must read spec_info via valid_mask (= arange(padded_bs) < real_bs).
    draft_input.topk_p = topk_p
    draft_input.topk_index = topk_index
    draft_input.hidden_states = hidden_states
    draft_input.padded_bs = topk_p.shape[0]   # 新字段, scheduler 用
    draft_input.real_bs = real_bs              # 新字段
```

#### 2. `eagle_draft_worker.py:draft_extend_for_decode` (line 333-345)
类似处理 refresh path 的 slice:
```python
# 当前:
if real_bs < padded_bs:
    topk_p = jax.lax.dynamic_slice_in_dim(topk_p, 0, real_bs, axis=0)
    ...
# Phase 4.10 改为: 不 slice, 保持 padded_bs
batch_output.next_draft_input.topk_p = topk_p           # padded
batch_output.next_draft_input.topk_index = topk_index   # padded
batch_output.next_draft_input.hidden_states = sliced_hidden  # 保持 padded
batch_output.next_draft_input.verified_id = batch_output.next_draft_input.verified_id[select_index]
batch_output.next_draft_input.padded_bs = padded_bs
batch_output.next_draft_input.real_bs = real_bs
# allocate_lens 仍 slice 回 real_bs (它不在 pytree, 不影响 jit; Phase 4.8c 已有)
```

#### 3. `eagle_util.py:EagleDraftInput`
- 加字段 `padded_bs: int = 0`, `real_bs: int = 0` (dataclass field)
- `tree_flatten` 把它们放 `aux_data` (Python int, 不进 leaf)

#### 4. `schedule_batch.py` 各处 spec_info 使用点
**关键**: scheduler 端不要再假设 `spec_info.topk_p.shape[0] == batch_size()`。改用 `spec_info.real_bs` 截取真实 row。

具体改动点 (用 grep `spec_info\.topk_p\|spec_info\.topk_index\|spec_info\.hidden_states\|spec_info\.verified_id` 找全):

- `schedule_batch.py:1591-1598` filter_batch 内 `info.spec_info.topk_p = info.spec_info.topk_p[keep_indices_jax]` —— `keep_indices_jax` 是 real-bs index 列表,在 padded array 上索引 → 拿到 real_bs 个真实 row。OK 不需改 (Python int index 即可)。但 result shape = real_bs 又破坏 padded — 需要 pad 回 padded_bs 或保持 real_bs (取决于下一步是否需要 padded)。
- 选项 A: filter_batch 后 spec_info 是 real_bs,下一次进 verify/refresh 前再 `padding_for_decode` pad → padded_bs。**这是当前流程**,只要 `padding_for_decode` 完整 pad spec_info 所有字段就 OK。检查是否 cover 所有 field。
- 选项 B: filter_batch 也保持 padded (mask out 被 filter 掉的 row)。更复杂。

**推荐选项 A** — 只要 `padding_for_decode` 已经 pad 所有 spec_info pytree leaves 即可。当前 `padding_for_decode` (eagle_draft_worker.py:400-503) 已 pad: `verified_id`, `topk_p`, `topk_index`, `hidden_states`, `seq_lens`, `allocate_lens`。**够了**!

#### 5. 验证 `merge_batch` (eagle_util.py:691-)
scheduler 把新 prefill spec_info concat 到 decode batch spec_info 时:
- decode batch spec_info 现在 padded (假设 padded_bs=8, real_bs=5)
- 新 prefill spec_info 是 real_bs=1
- 当前 `merge_batch` 用 `jnp.concatenate` → padded(8) + 1 = 9, scheduler 会以为 batch=9 → 错!

**这是 Phase 3.6 教训的核心**。修复:
```python
def merge_batch(self, spec_info: EagleDraftInput):
    if self.hidden_states is None:
        ... # 完整 copy spec_info
        return
    # 在 concat 前把 self 缩回 real_bs (用 host-side numpy 或 jax index)
    self.hidden_states = self.hidden_states[:self.real_bs]
    self.verified_id = self.verified_id[:self.real_bs]
    ...
    # 然后 concat (现在两边都是 real_bs)
    self.hidden_states = jnp.concatenate([self.hidden_states, spec_info.hidden_states])
    ...
    # 下一次 padding_for_decode 会重新 pad 到 padded_bs
```

`self.hidden_states[:self.real_bs]` 是 device-side slice (jax indexing),同样会 trace per real_bs。**这是个新 trace 源!**

**真正的 fix**: 用 `jnp.where` 或 mask 做 host-side numpy slice 后 `device_put`:
```python
hs_host = np.asarray(jax.device_get(self.hidden_states))[:self.real_bs]
self.hidden_states = device_array(hs_host, sharding=...)
```

但这又引入 host-device sync (Option A 我们否决的原因)。

**最优解**: 直接在 host 端管理 `hidden_states` 等 (numpy),只在进 jit 前 device_put 一次。但这是大重构。

#### 6. retract / abort 路径 (schedule_batch.py:1380~ retract_decode)
spec_info 在 retract 时也要正确处理 padding。当前用 device-side slice。如果保持 padded,retract 用 mask 不要 jit-slice。

| 风险点 | 影响 | mitigation |
|---|---|---|

## 关键陷阱 (新线程必读)

延续之前 HANDOFF_2026_05_18.md 的 6 大陷阱:

1. **不能简单去 real_bs slice** (Phase 3.6) — 但 Option B 就是要去。需要配套 scheduler 端 mask-aware 处理 merge_batch / filter_batch / retract。
2. **JAX 0.7+ sharded array `__getitem__`**: `arr[:real_bs]` ShardingTypeError,用 `jax.lax.dynamic_slice_in_dim`。但 `dynamic_slice` size 是 static → trace per real_bs。Option B 应改成 `jnp.where(mask, arr, 0)` mask 处理,保持 padded shape。
3. **spec_info 字段必须显式 NamedSharding(P())** (Phase 4.1)
4. **shape ≠ 脑补**: print 实际值,不靠推理
5. **closest-key 报告 ≠ 实际 miss**: chained 跨 mode 比对噪音常见。优先看 PHASE4.5b sub-jit counter 找真 miss
6. **pod editable install**: `kubectl cp` 单文件够,无需 reinstall

**新增陷阱 7 (本期教训)**:
- **`req.kv_committed_len` / `kv_allocated_len` 必须配对 update**: spec 流程下 prepare_for_decode bump allocated, `process_batch_result_decode` 内 spec 路径 bump committed,否则 `release_kv_cache` 内 `start_p == 0 == end_p`,over-alloc 永远不 free → KV leak (Issue #1147)
- **`EagleDraftInput.filter_batch` 必须同步 shrink `allocate_lens`**: 否则 batch shrink 后 `prepare_for_decode` assert 失败
- **`cur_allocate_lens` pad+restore 分离**: padding 用于 jit cache key 稳定,real_bs 给 scheduler。在 `next_draft_input.allocate_lens` 处 slice 回 real_bs (Phase 4.8c)
- **`allocate_lens` 不在 `EagleDraftInput.tree_flatten` children**: 改它不影响 jit cache key。Option B 关注真正 leaf (`hidden_states`/`topk_p`/`topk_index`/`verified_id`/`accept_length`)

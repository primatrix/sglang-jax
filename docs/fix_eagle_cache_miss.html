背景

开启 --speculative-algorithm EAGLE3 后,即使去掉 --disable-precompile,precompile 完成后首批请求仍触发大量 XLA 重编译。原因分两类:
  1. Precompile 覆盖不全 —— 多处 jit 边界没在 precompile 阶段被以全部 bucket 组合 trace 一次。
  2. EAGLE runtime padding 不一致 —— 部分张量按 real_bs 成形,或 pad 到非 bucket 值,导致即使输入 batch 命中 bucket,内部张量 shape 仍变化。
调试入口:在 ModelRunner.__init__ / scheduler 启动处加上 jax.config.update("jax_explain_cache_misses", True)(由 env var 控制),配合现有 tp_worker.py:418 处 count_pjit_cpp_cache_miss 包到 forward(...) 一并使用,就能复现每条 miss 的 (mode, bs, num_tokens, step_id) 指纹。

关键结论

  1. precompile 自己就漏了 464 项 —— SPEC_EXTEND 只对 bs=[16] 单值 trace(印证 R3),但模型 forward 同时还要 trace TARGET_VERIFY / DRAFT_EXTEND_AFTER_VERIFY 等多种 forward_mode + capture_hidden_mode 组合,这些组合在 precompile 循环里没穷举。
  2. 首请求 +22~48 次 miss —— 即便走 precompile 已覆盖的 bs/token,某些 (forward_mode, capture_hidden_mode, batch_size) pytree 静态 metadata 组合仍未命中。
  3. decode 稳态每 tick #cache_miss: 1 —— 长期存在的"每个 decode tick 都在重编译一次"现象。这是用户体感最差的部分(吞吐被压制),也是 F2/F4 必须根除的核心目标。
  4. SPEC_DECODE precompile 229s —— 即使只 4 个 bs 桶,内部完整链路 trace 已经 ~57s/桶,F2 加上 step_id × bs × token 矩阵后启动时间会显著上升,需要在做 F2 时同步关注。

Padding 契约(验收标准)

所有 spec 路径(extend / decode / draft 多步 / verify)在进入任何 jit 边界前,必须把 runtime 形状向上取整到最近的 precompile bucket。precompile bucket 是「契约值」,runtime 不允许产生 bucket 外的 shape。

维度输入定义域映射规则示例 (bucket = [1, 8, 16])
batch size 1 ≤ real_bs ≤ max_bs_bucket padded_bs = min{ b ∈ bs_buckets : b ≥ real_bs } 5 → 8; 9 → 16; 16 → 16
prefill token 数 1 ≤ num_tokens ≤ max_token_bucket padded_tokens = min{ t ∈ token_buckets : t ≥ num_tokens } (bucket=[64,128,256,512]) 100 → 128; 500 → 512
cache_loc 由 padded_bs 决定 查表 cache_loc_buckets[bs_idx](含 spec 增量,见 F7) —
tree mask max_context_len 由 max(seq_lens) 决定 同 prefill token 桶向上取整,无 fallback(见 F5) —
越界处理:real_bs > max_bs_bucket 或 num_tokens > max_token_bucket 时,直接报错并提示扩大 --precompile-bs-paddings / --precompile-token-paddings。禁止静默 fallback 到「向上取下一个 2 的幂」之类的非 bucket 值 —— 这正是 R6 的 bug 模式。
对齐非 MTP 路径的现有做法(无需另造轮子,直接复用):
  • 统一的查桶函数:common_utils.py:42 pad_to_bucket(value, buckets) —— 内部 bisect.bisect_left 找最小 ≥ value 的 bucket;idx >= len(buckets) 直接 raise ValueError("No bucket >= {value}. Available: ... Increase bucket sizes or max capacity.")。这就是非 MTP 的「越界禁止 fallback」实现。 
    def pad_to_bucket(value, buckets):
    idx = bisect.bisect_left(buckets, value)
    if idx < len(buckets):
        return buckets[idx], idx
    raise ValueError(f"No bucket >= {value}. Available: {buckets}. ...")
  • buckets 一定包含 max 哨兵:compilation_manager.py:75-92 _compute_token_buckets / _compute_bs_buckets 在末尾必定追加 max_padded_num_tokens / max_padded_batch_size,确保用户传入的最大配置一定是某个 bucket,real_bs ≤ max_bs_bucket 永远能命中。
  • 调度侧统一调用点:schedule_batch.py:1747-1748,1922 已经在调度合批时调用 pad_to_bucket(total_bs, bs_paddings) / pad_to_bucket(max_tokens, token_paddings) —— spec 路径要复用同一函数与同一 bucket 列表(model_runner.compilation_manager.bs_buckets / token_buckets),不得另起一套。

因此 F3 / F4 / F5 的「向上取整」具体落地 = 在 spec 入口处调用 pad_to_bucket,使用与非 MTP 完全相同的 buckets 与 raise 行为,产生 bucket 外形状这条路径在代码层面就不存在。

验收条件(F1–F8 全部完成后必须满足):
  1. 给定 --precompile-bs-paddings 1 8 16:对 real_bs ∈ {1..16} 的任意 decode 请求,首次 forward 的 cache_miss_count = 0。
  2. 给定 --precompile-token-paddings 64 128 256 512 1024:对 num_tokens ∈ [1, 1024] 的任意 prefill 请求,首次 forward 的 cache_miss_count = 0。
  3. SGLANG_JAX_EXPLAIN_CACHE_MISSES=1 下整个压测周期无 jax_explain_cache_misses 日志输出。
  4. 同时混合多种 bs 与 prompt 长度的并发场景下,稳定后 cache_miss_count 始终为 0。

Precompile 三分法:EXTEND / DECODE / RUNTIME

对 spec 路径而言,需要 precompile 的不止是「模型 forward」一个 jit。把所有 device 上要跑的 jit 分成三类:
  1. EXTEND —— prefill 时进入 device 的所有 jit(target extend forward、draft extend forward、capture 后的 topk)。
  2. DECODE —— decode tick 时模型 forward 类 jit(draft 多步 forward、target verify forward、draft refresh forward)。
  3. RUNTIME —— forward 之外、在 decode tick 内被调用的所有辅助 jit / Pallas 内核(tree 结构构建、verify 树采样、topk_probs_from_logits、select_top_k、update_eagle_lists、采样器、renorm 等)。这一类是非 MTP 模式没有、最容易被忽略的部分。
三类都需要按 (bs 桶, seq_len 桶) 笛卡尔积 precompile(部分边界还要叠加 step_id 维度,见 边界清单)。
核心不变量(precompile 完成后必须始终成立):
MTP 运行过程中,任何一个在 device 上执行的函数,其每一个输入张量的 shape 都必须命中已经 precompile 过的 cache key。
这是验收的终极硬约束。任何打破这一不变量的代码路径都视为 bug。

接受率(accept_lens)导致的 shape 漂移

spec 流程里有一类 shape 不是固定的,而是由 verify 阶段的接受率决定 —— 主要是 verified_id、extend_seq_lens、以及由它们派生的 cu_q_lens / cu_kv_lens / page_indices。这是 RUNTIME / DECODE 类 cache miss 的最大来源。

padding 处理原则(F4 落地细则):

  • 形状统一按 padded_bs × draft_token_num(verify 输出固定 leading dim);
  • accept_lens 在 padding 行填 0,真实行保留 1..N;
  • 需要按 accept_lens 切片的位置(select_index、extend_seq_lens)改为在 padded_bs 维度上构造,padding 行取 dummy 索引,保证 gather 合法且结果不被消费;
  • 「padding token 不申请 KV cache」继续遵守 —— page_indices 在 padding 行复用 dummy page,确保 shape 命中桶且不污染真实 KV。

  2. D2 draft refresh 的 cu_q/cu_kv/page_indices 形状取决于 accept_lens;F4 要求 padded_bs 化后,accept_lens 在 padding 行填 0、保留 (padded_bs, draft_token_num) 形状一致。
  3. C2 / E7 的 max_context_len:R6 fallback 必须去掉,改严格 pad_to_bucket。
  4. B3 / E2 的 step_id 静态键:N 项 cache / bs 桶不可避免,但 precompile 必须显式遍历 step_id ∈ [0, num_steps),否则首请求即逐步 miss。
  5. B5 的 TARGET_PADDING=16384:F6 把它移入 compilation_manager.cache_loc_paddings,与 bs 桶联动。
  6. C5/C6/E10 非 greedy 路径:precompile_spec_decode 只构造 greedy dummy,生产首次出现 top-p/top-k>1 请求时冷编译;F2 必须为「sampling_info 的代表性组合」额外补 1~2 条 trace。

根因清单

R1. 开启 spec 后,target 模型 precompile 被完全跳过 覆盖

scheduler.py:433-441 在 spec_algorithm 非 None 时只跑 draft_worker.run_spec_decode_precompile(),标准 tp_worker.run_precompile() 不执行。

  • target extend 只在 bs = max_padded_batch_size, token_bucket=* 这一条线上被 trace (eagle_worker.py:283-313);
  • target decode (verify) 只在 (bs, draft_token_num) 一组上被 trace (eagle_worker.py:315-366);
  • 标准 _precompile_decode 的 bs × bs decode 矩阵完全没跑。

R2. Draft 模型 precompile 覆盖不全 覆盖

precompile_spec_decode 对每个 bs 只调用一次完整链路,但 draft 多步循环中 每个 step_id 对应一组独立的 attention metadata(flashattention_backend.py:335-379);循环只跑一次 → step 0..N-1 都只在该 bs 下被 trace,其它 bs bucket 完全没跑过完整 N 步循环。

另外,draft 底层 ModelWorker.compilation_manager 存在,但 run_precompile() 在 spec 路径下被绕过 —— draft forward 自身的 token/bs 桶完全没 trace。

R3. Extend 阶段 spec 侧无 bs padding padding

eagle_worker.py:80-111 显式注释 # FIXME(pc) add padding logic here。prefill 真实 bs 直接进入 target extend 和 draft_extend_for_prefill,而 R1 已说明 target extend 只在 bs = max_padded_batch_size precompile,真实 bs 不命中 → 重编译。

R4. draft_extend_for_decode 输入按 real_bs 切片,非 padded_bs padding

verify 在 eagle_worker.py:159-204 产出的 verified_id / hidden_states 形状为 padded_bs*(spec_steps+1),但下游:

  • select_index 由 real_bs + accept_lens[: real_bs] 构造 (eagle_draft_worker.py:256-261);
  • rep_logits[select_index] 得到 real_bs 长度,送入 jitted topk_probs_from_logits;
  • extend_seq_lens = full(bs_real, step_plus_1) 中的 bs 是 real_bs。

同一 padded_bs 下,real_bs 不同 → shape 不同 → 重编译。

R5. padding_for_decode input_ids 重置 需复核

eagle_draft_worker.py:387-388 末尾把 input_ids / positions 重置为 np.empty(bs * topk, np.int32)。topk 在 precompile 中只取 CLI 配置那一个值;若运行时 topk 动态变化才会有问题(当前看是静态,作为「确认无问题」纳入复核)。

R6. build_tree_kernel_efficient max_context_len 兜底非 bucket padding

eagle_draft_worker.py:451-457 _pick_context_len 当 max_seq_len 不在 precompile_token_paddings 内,fallback 1 << (max_seq_len-1).bit_length(),该值未必在 bucket 集合内 → 新 cache 项。

R7. TARGET_PADDING = 16384 硬编码 魔法值

flashattention_backend.py:373:# FIXME Handle padding, this will be move to precompile。当 bs * cdiv(max_seq_len + spec_steps*topk, page_size) > 16384 时不再 pad,shape 由真实值决定 → miss。

R8. cache_loc / page_indices 桶未覆盖 spec 增量 覆盖

_compute_cache_loc_buckets(common_utils.py:95-97)按 bs_bucket * ceil_to_page(max_req_len) 算,只在标准 decode 场景下成立。EAGLE verify / draft 多步会消耗额外 page,最大可能 cache_loc 应额外加 (spec_steps+1)*draft_token_num * bs 的余量。

修复策略

F1. 标准 precompile 在 spec 模式下也跑 → R1

scheduler.py:433-441:无论是否启用 spec,都先执行 tp_worker.run_precompile(),再执行 draft_worker.run_spec_decode_precompile()。仅延长启动时间,无精度影响。

F2. Spec precompile 覆盖完整 (bs × step) 矩阵 → R2

重写 EAGLEWorker.precompile_spec_decode:外层遍历 precompile_bs_paddings,内层显式 trace 每个 step 的 draft forward 与 verify。precompile_spec_extend 改为遍历 (bs_bucket, token_bucket) 而非只 (max_padded_bs, token_bucket)。

F3. Extend 阶段加 spec bs padding → R3

实现 eagle_worker.py:80-111 FIXME:prefill 进入 spec 路径前,按 Padding 契约 向上取整把 model_worker_batch 的 seq_lens / input_ids / positions / cache_loc / extend_seq_lens 对齐到 (padded_bs, padded_tokens) 双桶;padding token 不申请 KV cache。

F4. draft_extend_for_decode 全程 padded_bs 化 → R4

所有按 real_bs 切片的位置改为 padded_bs(padded_bs 由 契约 ceil 到 bs_buckets):select_index 在 padded_bs 维度上构造,padding 行 accept_lens 填 0,select_index 取固定 dummy 行;extend_seq_lens 长度 = padded_bs,padding 位填 0(padding token 不申请 KV)。

F5. build_tree_kernel_efficient 严格 bucket 化 → R6

_pick_context_len 去掉 bit_length fallback,改为「bisect_left 到下一个 precompile_token_paddings 桶,超出则报错」,确保只产生 bucket 内的 shape。

F6. 把 TARGET_PADDING 移入 precompile 配置 → R7

compilation_manager 计算 cache_loc_paddings 时加入 spec 增量;get_eagle_multi_step_metadata 中 TARGET_PADDING 改由 model_runner.compilation_manager.cache_loc_paddings[bs_idx] 提供,消除魔法常数 16384。

F7. cache_loc / page_indices bucket 含 spec 余量 → R8

cache_loc_bucket = bs_bucket * ceil_to_page(max_req_len)
                 + bs_bucket * ceil_to_page((spec_steps + 1) * draft_token_num)

确保所有 verify / 多步 draft 的 cache_loc 都落在已 trace 形状内。

F8. cache-miss 诊断开关 → 全部

  • env var SGLANG_JAX_EXPLAIN_CACHE_MISSES=1 时,ModelRunner.__init__ 调用 jax.config.update("jax_explain_cache_misses", True);
  • 把 count_pjit_cpp_cache_miss 包裹范围从 sample(...) 扩展到 tp_worker.forward_batch_generation 全 forward;
  • 落实 compilation_manager.py:329 register_variant_if_new 调用点,首次出现的 variant key 打日志。

cache_miss_probe —— 包一个 contextmanager,内部启用 explain + 用 jtu.count_pjit_cpp_cache_miss 计数,退出时返回 miss 数: 
from contextlib import contextmanager
import jax
from jax._src import test_util as jtu

@contextmanager
def cache_miss_probe(label):
    prev = jax.config.read("jax_explain_cache_misses")
    jax.config.update("jax_explain_cache_misses", True)
    with jtu.count_pjit_cpp_cache_miss() as count:
        yield count
    jax.config.update("jax_explain_cache_misses", prev)
    print(f"[{label}] cache_miss={count[0]}")
所有 L2~L4 用例用 assert count[0] == 0, ... 断言。

  • 模型层数固定 1~2 层,hidden=128,num_attn_heads=4;
  • 权重 jnp.zeros 或随机小张量;
  • page_size、num_pages 用最小可工作值;
  • compilation_manager 沿用真实实现,但 max_padded_batch_size / max_padded_num_tokens 设小(如 16 / 1024)。

对每个 RUNTIME jit 写一个 test_no_miss_after_warmup:

  1. 先用一组 dummy 输入(命中预期 bucket)调用一次 —— 触发编译;
  2. 在 cache_miss_probe 中,用 同 bucket 但 value 不同、以及 不同 bucket 的多组输入调用;
  3. 断言「同 bucket 不同 value 的调用」miss=0;
  4. 断言「不同 bucket 但属预期遍历集」的调用,在补跑那些 bucket 的 warmup 后再次为 0。

L4 测试(stub 模型 + 全链路)

  1. StubModelRunner 初始化 → 跑 spec precompile(F1+F2 后的版本)。
  2. 构造 50 个混合请求(随机 prompt 长度 1~max_token_bucket、随机生成长度),用 spec 完整链路并发处理。

调试技巧

  • L2 失败时,在 probe 内同时开 JAX_LOG_COMPILES=1,日志会打印每次编译的 jaxpr,直接定位是哪个 static arg / shape 变了。
  • L3 怀疑某个 bucket 漏 trace,把 compilation_manager.cache_loc_buckets 与 bs_buckets 打到日志头,运行时把命中桶 index 也打印,对比就能立刻发现。
  • 所有 L 用例统一通过环境变量 SGLANG_JAX_EXPLAIN_CACHE_MISSES=1 一键启用 explain 日志,避免散落配置。

待澄清

  • draft_model_runner.forward 的 jit 边界 / cache key 细节未读;F2 实施前需要确认它确实按 (num_tokens, bs) 形状变化做 cache key,否则需要在 model 层补 static_argnames。
  • 是否所有调用都已确保「padding token 不申请 KV cache」需要随 F3/F4 联动复核。
  • StubModelRunner 的最小接口面:ModelRunner 依赖较重(权重 / KV pool / sharding / attention backend),L3/L4 写起来前需要一次小 spike,先确认能用 hidden=128、1~2 层、最小 num_pages 起一个跑通 precompile 的 mock;否则 L3/L4 可能阻塞。

设计决策(已敲定)

  1. C3 verify custom_mask 桶化:不再 pad 到最大值,复用 EXTEND 一样的桶思路 —— 新增一份 custom_mask_buckets(基于 precompile_token_paddings 的子集),运行时把 custom_mask 长度通过 pad_to_bucket ceil 到最近 bucket。precompile 时遍历 (bs 桶) × (custom_mask 桶)。
  2. RUNTIME 独立遍历:在 precompile_spec_decode 末尾 直接 import 每个 jit / Pallas wrapper 手动调用,不走完整 spec 链路。对 B3 / E2 (update_eagle_lists) 等 score_list/token_list/parents_list 随 step_id 变化的形状,通过 mock 构造 dummy 张量并 ceil 到合适 bucket。
  3. 系统无 reserved padding page:F4 中 padding 行的 page_indices 不取 reserved page,改用其它合法 dummy 策略(例如复用 batch 内首个真实 page,确保 gather 合法且不写入)。
  4. L5 StubModelRunner:暂未敲定,移至「待澄清」继续跟。
  5. 采样路径:本期只验证 greedy。C5/C6 非 greedy precompile 推迟到下一期,文档保留但不进 L2 矩阵。
  6. Precompile 启动耗时:本期不优化,先以「基线测量」为准。
  7. Spec EXTEND bs 对齐策略(2026-05-18 新增):走「方案 B — precompile 端覆盖用户输入的所有 bs 桶 + runtime 端 ceil pad」。
    • precompile 端:precompile_spec_extend 改为 product(user_bs_buckets, token_buckets),遍历用户通过 --precompile-bs-paddings 传入的所有 bs 桶,而非只 [max_bs]。
    • runtime 端:get_spec_model_worker_batch 在 EXTEND 分支调 pad_to_bucket(real_bs, user_bs_buckets),把 seq_lens / input_ids / positions / cache_loc 等 leading dim 抬到 ≥ real_bs 的最近桶。

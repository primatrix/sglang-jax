# MiMo-V2-Flash FP8 Quantization Benchmark Results

- Model: MiMo-V2-Flash (`hidden_size=4096, num_attention_heads=64, num_key_value_heads=4/8, head_dim=192, v_head_dim=128`)
- Quantization: FP8 (float8_e4m3fn), weight-only (no activation quantization)
- Kernel: BF16 和 FP8 per-channel 使用 `lax.dot_general`（XLA 原生），FP8 block-128 使用 Pallas `blockwise_kernel`
- Platform: TPU v6e (2x2, 4 chips, single-host)
- 计时方式: 从 JAX profiler trace 中提取 kernel-level `device_duration_ps`，匹配 `tf_op` 中的 scope name

---

## HLO 观察到的实际 Quantized Matmul Cases

以下 3 个 case 来自模型实际编译产生的 HLO，均为 TP=1、block-128 场景：

| HLO Kernel | 计算 (X x W^T -> Out) | 对应 Projection | BF16 (ms) | FP8 (ms) |
|---|---|---|---|---|
| `quantized_matmul_kernel_128_256_128` | `bf16[2048, 4096] x f8e4m3fn[768, 4096]^T -> bf16[2048, 768]` | k_proj (full attn, tokens=2048) | 0.020 | 0.706 |
| `quantized_matmul_kernel_128_256_128` | `bf16[2048, 4096] x f8e4m3fn[1024, 4096]^T -> bf16[2048, 1024]` | v_proj (SWA, tokens=2048) | 0.025 | 0.941 |
| `quantized_matmul_kernel_64_256_128` | `bf16[64, 4096] x f8e4m3fn[768, 4096]^T -> bf16[64, 768]` | k_proj (full attn, tokens=64) | 0.002 | 0.004 |

> kernel 名中的数字为 Pallas tiling 参数 (block_m, block_n, block_k)。Weight 以转置形式 `[out, in]` 存储。

---

## 结论

**FP8 block-128 在 TPU v6e 上全面落后于 BF16**，根因是 `tuned_block_sizes.py` 中 TPU v6 仅有 INT8 调优条目，没有 FP8 条目，Pallas blockwise kernel 使用未调优的默认 block sizes。

**FP8 per-channel 性能接近 BF16**，因为 per-channel 走的是 `lax.dot_general`（XLA 原生算子），与 BF16 使用相同的 MXU 路径。大 batch (tokens >= 512) 时 per-channel FP8 与 BF16 基本持平，部分 case 略快 (1.02x-1.12x)；小 batch 时 FP8 有额外的 dequant 开销，比 BF16 慢。

| 观察 | 说明 |
|------|------|
| BF16 TFLOPS 可达 500-3200 | `lax.dot_general` 充分利用 MXU，接近 TPU v6e 峰值 |
| FP8 per-channel 大 batch 接近 BF16 | 同样走 `lax.dot_general`，大矩阵时 dequant 开销可忽略 |
| FP8 per-channel 小 batch 慢于 BF16 | dequant 开销在小矩阵中占比较大 |
| FP8 block-128 TFLOPS 封顶 ~18 | 未调优 Pallas kernel 吞吐极低 |
